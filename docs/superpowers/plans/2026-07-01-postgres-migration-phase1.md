# Postgres Migration Phase 1 (Engine Swap) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Switch the backend's Prisma datasource from MySQL to PostgreSQL, with no schema shape changes, no application code changes, and no data migrated from the existing MySQL database.

**Architecture:** Change the `provider` in `schema.prisma` and `DATABASE_URL` in `.env` to point at the local Postgres instance and the pre-created `tms` database. Because Prisma's generated migration SQL is provider-specific, the existing MySQL-flavored migration history cannot replay against Postgres — it is deleted and regenerated as a single fresh `init` migration. The old MySQL database and its data are left completely untouched; rollback is a two-line config revert.

**Tech Stack:** NestJS 11, Prisma 6 (MySQL → PostgreSQL), bcrypt, pnpm.

## Global Constraints

- Do not modify, export, or delete anything in the existing MySQL database (`thesis_management`). It stays as-is, unused, as a fallback.
- No changes to `schema.prisma` model definitions, field types, or relations — only the `datasource` block changes.
- No changes to files under `backend/src/` — verified during brainstorming that no MySQL-specific SQL or type annotations are in use.
- Local Postgres 18 is already running (`postgresql-x64-18` service, port 5432) and the `tms` database already exists — no install/setup steps in this plan.

### Task 1: Switch the Prisma datasource to PostgreSQL

**Files:**
- Modify: `backend/prisma/schema.prisma:6`
- Modify: `backend/.env` (not committed — gitignored)

**Interfaces:**
- Produces: `DATABASE_URL` env var consumed by Prisma at `backend/prisma/schema.prisma:7` (`url = env("DATABASE_URL")`) — unchanged reference, only the value changes.

- [ ] **Step 1: Change the datasource provider**

In `backend/prisma/schema.prisma`, change line 6:

```prisma
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

- [ ] **Step 2: Update the connection string**

In `backend/.env`, replace the `DATABASE_URL` line with:

```
DATABASE_URL="postgresql://postgres:<your-postgres-password>@localhost:5432/tms?schema=public"
```

Replace `<your-postgres-password>` with the actual `postgres` superuser password.

- [ ] **Step 3: Commit the schema change**

`.env` is gitignored, so only `schema.prisma` is committed.

```bash
git add backend/prisma/schema.prisma
git commit -m "chore(db): switch prisma datasource from mysql to postgresql"
```

### Task 4: Bootstrap the first admin user

**Files:** None in the repo — this is a one-time manual DB insert via DataGrip.

**Interfaces:**
- Consumes: `users` table schema from Task 2 (`id`, `username`, `password_hash`, `role`, `is_active`, `created_at`, `updated_at`, `refresh_token`).
- Produces: one `User` row with `role = 'ADMIN'`, usable to log in via `POST /auth/login`.

- [ ] **Step 1: Generate a bcrypt hash for the chosen admin password**

From `backend/`, run (replace `YOUR_PASSWORD` with the real password):

```bash
node -e "require('bcrypt').hash('YOUR_PASSWORD', 10).then(h => console.log(h))"
```

Expected output: a single line starting with `$2b$10$...` — copy this hash.

- [ ] **Step 2: Insert the admin row via DataGrip**

In DataGrip, connected to the `tms` database, run in the SQL console (replace `<hash>` with the value from Step 1, and choose a username):

```sql
INSERT INTO users (username, password_hash, role, is_active, updated_at)
VALUES ('admin', '<hash>', 'ADMIN', true, CURRENT_TIMESTAMP);
```

Note: `updated_at` must be set explicitly. It's `NOT NULL` with no SQL-level default — `@updatedAt` in `schema.prisma` is populated by the Prisma Client at the application layer, not by a database default, so any write that bypasses the Prisma Client (like this raw insert) must supply it. `created_at` doesn't need this since it has `DEFAULT CURRENT_TIMESTAMP` in the DDL.

- [ ] **Step 3: Verify login works**

With the backend running (from Task 3), run:

```bash
curl -i -X POST http://localhost:3000/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"YOUR_PASSWORD"}'
```

Expected output: `HTTP/1.1 201 Created` (or `200 OK` depending on NestJS default), with a JSON body containing an access token, and a `Set-Cookie` header for the refresh token.

- [ ] **Step 4: Confirm rollback path still works (optional but recommended)**

To confirm the rollback plan from the spec is real, temporarily revert **both** `backend/prisma/schema.prisma`'s `provider` (back to `"mysql"`) **and** `backend/.env`'s `DATABASE_URL` (back to the MySQL value), then restart `pnpm run start:dev` — it should boot cleanly against the untouched `thesis_management` MySQL database. Reverting `.env` alone is not sufficient: Prisma's generated client is tied to the `provider` declared in `schema.prisma` at generate time, so an env-only swap fails with a `P1012` provider-mismatch error even though the connection string itself is valid MySQL. Then switch both back to the Postgres `provider`/`tms` `DATABASE_URL` before continuing any further work.
